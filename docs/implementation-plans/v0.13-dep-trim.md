# v0.13 Dependency Trim Plan

**Last updated:** 2026-05-01
**Tracking issue:** [#62](https://github.com/padamson/playwright-rust/issues/62)
**Targets:** v0.13.0

## Goal

Reduce cold-build time and the binary-size footprint we impose on
downstream consumers, without changing the public API. Issue #62 reports
a ~30-minute first-build for `playwright-rs = "0.12"` as a dev-dep on
Apple Silicon. We can do better.

## Scope

This document is the analysis-and-recommendation phase. Each "Land
this" item below corresponds to a sub-issue tracked separately and
landed as its own PR. CHANGELOG entries accumulate under
`[Unreleased]` until v0.13.0 is cut.

## Direct dependencies, ranked by impact

| Crate | Current | Used in | Savings opportunity | SemVer impact | Recommendation |
|---|---|---|---|---|---|
| `image` | always-on, png-only | `assertions.rs` (screenshot diff/baseline write) | Behind `screenshot-diff` feature flag. Only consumers using `to_have_screenshot()` need it. | Removing from default = breaking. Adding flag and keeping default = compatible. | **Land this**: ship as opt-out with `screenshot-diff` in `default = [...]`, with a deprecation note that v1.0 may flip the default. Compatible patch. |

## Build dependencies (`build.rs`)

| Crate | Used for | Recommendation |
|---|---|---|
| `reqwest` (`features = ["blocking"]`) | Driver download from Azure CDN | **Replace with `ureq`**. `reqwest` pulls in hyper, tower, h2, etc. — a massive build-time tree we use exactly once at build time, in blocking mode, for one HTTP GET. `ureq` is a thin synchronous client (~5 deps total). Build-only change, zero user-facing impact. |
| `zip` | Driver archive extraction | Already minimal. Skip. |
| `dirs` | Cache directory location | Tiny. Skip. |

## Tokio feature audit

`tokio = "full"` enables 13 sub-features. Audit of every `tokio::` call
site in `crates/playwright/src/`:

| Call site (representative) | Required tokio feature |
|---|---|
| `tokio::fs::*` (read, write, create_dir_all, remove_file) | `fs` |
| `tokio::io::{split, duplex}` | `io-util` |
| `tokio::net::*` (transitively via `tokio-tungstenite`) | `net` |
| `tokio::process::Command` | `process` |
| `tokio::runtime::*` (Handle, Runtime in tests/examples) | `rt` |
| `tokio::spawn`, `block_in_place` | `rt-multi-thread` |
| `tokio::main`, `tokio::test` macros | `macros` |
| `tokio::signal::ctrl_c` | `signal` |
| `tokio::sync::{mpsc, oneshot}` | `sync` |
| `tokio::time::{sleep, timeout}` | `time` |
| `tokio::select!`, `tokio::join!` | `macros` (already required) |

**Minimal set:** `fs`, `io-util`, `macros`, `net`, `process`, `rt`,
`rt-multi-thread`, `signal`, `sync`, `time` — 10 features.

`full` adds 3 we don't use: `parking_lot` (we depend on it directly,
don't need tokio's integration), `tracing` (tokio internal
instrumentation; we use `tracing::` directly elsewhere, not tokio's
self-instrumentation), `io-std` (`tokio::io::stdin/stdout/stderr`
wrappers).

Switching to the explicit set has minor build-time savings — the major
benefit is documenting what we actually use so future audits are easy.

## Ecosystem precedent

Brief survey of comparable browser-automation crates:

- **`chromiumoxide`** — granular runtime flags (`tokio-runtime`,
  `async-std-runtime`), browser-flavour flags (`firefox`,
  `experimental-firefox`). More aggressive than our needs; we're
  tokio-only.
- **`fantoccini`** — narrow tokio features, optional `tracing`,
  `chrome` / `firefox` flags for browser-specific helpers.
- **`headless_chrome`** — narrow tokio, `fetch`, `nightly` flags.

**Convention:** opt-out flags for "every consumer" features, opt-in
flags for "some consumers" features. `image` for screenshot-diff is
clearly the latter.

## SemVer summary

The proposed changes split into three categories:

1. **Compatible (patch-eligible):** narrow `tokio` features, swap
   `reqwest` → `ureq` in `build.rs`, replace `mime_guess`. Could land
   on the v0.12.x line.
2. **Compatible-via-default (patch-eligible):** add `screenshot-diff`
   feature, **keep it in `default = [...]`** so existing consumers see
   no break. Document that v1.0 will likely flip the default.
3. **Compatible-with-default-flip (minor break):** in v0.13.0,
   optionally remove `screenshot-diff` from default features so
   downstream cold builds shrink without an explicit opt-out. This is
   the larger user-facing change and the natural anchor for a 0.13.0
   release.

## Sub-issues (one PR each)

To be opened as we proceed:

1. Replace `reqwest` with `ureq` in `build.rs`
2. Narrow `tokio` features from `"full"` to the explicit minimal set
3. Replace `mime_guess` with a small hand-rolled extension-to-MIME map
4. Add `screenshot-diff` feature flag (default-on for compatibility)
5. (v0.13.0 anchor) Flip `screenshot-diff` to default-off, document the
   breaking change in release notes

Items 1–4 are individually compatible patches. Item 5 is what
actually justifies the 0.13.0 minor bump.

## Out of scope

- Replacing `regex`, `url`, `glob`, `parking_lot`, `tracing`,
  `serde*`, `thiserror`, `async-trait`, `futures-util`, `base64`,
  `libc` — analysed above, not worth it.
- Optional async-runtime support (e.g. `async-std`) — chromiumoxide
  does this; we have no demand and it doubles maintenance surface.
- `no_std` support — Playwright requires a real OS, not a target.
